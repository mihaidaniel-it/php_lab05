#  Лабораторная работа №5

## Объектно-ориентированное программирование в PHP

---

##  Цель работы

Целью данной лабораторной работы является освоение принципов объектно-ориентированного программирования (ООП) в PHP. В рамках работы были изучены:

* создание классов и объектов;
* инкапсуляция данных;
* разделение ответственности между классами;
* использование интерфейсов для построения гибкой архитектуры;
* работа с коллекциями объектов;
* генерация HTML с помощью PHP.

---

##  Условие задачи

Необходимо было разработать приложение для управления банковскими транзакциями, которое позволяет:

* хранить транзакции;
* добавлять и удалять транзакции;
* искать транзакции по ID;
* выполнять вычисления (суммы, фильтрация);
* сортировать данные;
* отображать транзакции в виде HTML-таблицы.

Решение должно быть реализовано с использованием ООП.

---

##  Структура решения

В процессе выполнения работы были реализованы следующие компоненты:

### 1. Класс `Transaction`

Описывает одну банковскую транзакцию.

**Содержит:**

* ID транзакции;
* дату;
* сумму;
* описание;
* получателя.

**Особенности:**

* все свойства являются приватными;
* доступ к данным осуществляется через getter-методы;
* реализован метод `getDaysSinceTransaction()` для вычисления количества дней с момента транзакции.

---

### 2. Интерфейс `TransactionStorageInterface`

Определяет контракт для работы с хранилищем транзакций.

**Методы:**

* добавление транзакции;
* удаление по ID;
* получение всех транзакций;
* поиск по ID.

---

### 3. Класс `TransactionRepository`

Реализует интерфейс хранения данных.

**Функции:**

* хранение массива транзакций;
* добавление и удаление транзакций;
* поиск и получение данных.

**Особенность:**

* инкапсуляция — доступ к данным только через методы.

---

### 4. Класс `TransactionManager`

Отвечает за бизнес-логику приложения.

**Реализованные функции:**

* расчет общей суммы транзакций;
* расчет суммы за выбранный период;
* подсчет транзакций по получателю;
* сортировка по дате;
* сортировка по сумме.

**Важно:**

* не хранит данные, а использует репозиторий;
* работает через интерфейс, что делает архитектуру гибкой.

---

### 5. Класс `TransactionTableRenderer`

Отвечает за отображение данных.

**Функция:**

* преобразует массив транзакций в HTML-таблицу.

**Особенности:**

* строго разделяет логику и представление;
* реализован как `final` класс.

##  Использованные принципы ООП

### Инкапсуляция

Данные скрыты внутри классов и доступны только через методы.

### Абстракция

Интерфейс определяет поведение без реализации.

### Полиморфизм

`TransactionManager` работает с интерфейсом, а не с конкретной реализацией.

### Разделение ответственности (SRP)

Каждый класс выполняет строго одну задачу:

* модель,
* хранилище,
* логика,
* отображение.

---

##  Ответы на контрольные вопросы

**1. Зачем нужна строгая типизация?**
Позволяет избежать ошибок типов, делает код более надежным и предсказуемым.

**2. Что такое класс?**
Класс — это шаблон для создания объектов, содержащий свойства и методы.

**3. Что такое полиморфизм?**
Это возможность использовать один интерфейс для разных реализаций.

**4. Что такое интерфейс?**
Это контракт, который определяет, какие методы должен реализовать класс.

**5. Преимущества интерфейсов**
Позволяют легко менять реализацию (например, заменить массив на базу данных) без изменения бизнес-логики.
